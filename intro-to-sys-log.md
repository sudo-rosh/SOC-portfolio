## What is a log?

A log is a record of events within a system. Regardless of log type, a single log entry typically contains:

- A **timestamp** of when the event occurred
- The **system or application name** that generated the entry
- The **type of event** that occurred
- **Additional context** — e.g. the user who triggered the event, or the IP address involved

These four data points are the baseline checklist I use when reading any unfamiliar log: timestamp, source, action, normal-or-suspicious.

---

## Types of logs

| Log type | What it captures |
|----------|------------------|
| **Application logs** | App-specific status, errors, warnings |
| **Audit logs** | Operational activity relevant to regulatory compliance |
| **Security logs** | Logins, permission changes, security-relevant activity |
| **Server logs** | System, event, error, and access logs generated by a server |
| **System logs** | Kernel activity, system errors, boot sequences, hardware status |
| **Network logs** | Traffic, connections, network-related events |
| **Database logs** | Queries, updates, database activity |
| **Web server logs** | Requests processed by a web server — URLs, response codes |

## Log formats — Semi-structured logs

Semi-structured logs mix predictable structured fields with free-form unstructured text. The most common example is **Syslog**, a widely adopted logging protocol for system and network logs.

### Worked example — parsing a real syslog entry

```
May 31 12:34:56 WEBSRV-02 CRON[2342593]: (root) CMD ([ -x /etc/init.d/anacron ] && if [ ! -d /run/systemd/system ]; then /usr/sbin/invoke-rc.d anacron start > /dev/null; fi)
```

Breaking this down piece by piece:

| Fragment | Meaning |
|----------|---------|
| `May 31 12:34:56` | Timestamp |
| `WEBSRV-02` | Hostname — the system that generated the entry |
| `CRON[2342593]` | Process name and PID that logged the event |
| `(root)` | User context the command ran under |
| `[ -x /etc/init.d/anacron ]` | Conditional check: does the file exist **and** is it executable? (`-x` tests both) |
| `&&` | Logical AND — only proceed to the next command if the previous check was true |
| `[ ! -d /run/systemd/system ]` | Conditional check: does this directory **NOT** exist? |
| `/usr/sbin/invoke-rc.d anacron start` | The actual command executed — starts the Anacron service |

**Plain-English summary:** CRON, running as root, checked whether Anacron was installed and executable. If the system was *not* running under systemd, it started the Anacron service and discarded the output.

---

## Why this matters for SOC / DFIR work

- This single log line demonstrates why log reading is a *skill*, not a lookup — the meaning isn't in any one field, it's in how the fields chain together (the conditional logic here only makes sense read left to right as a sequence)
- Syslog's semi-structured nature is exactly why raw log reading practice matters more than memorizing a format spec — real logs mix predictable fields with command syntax, flags, and conditional logic that has to be parsed contextually
- The four-question framework (timestamp / source / action / normal-or-suspicious) applied to this entry: this is *normal* scheduled system behavior (Anacron startup check), not an indicator of compromise — but the same parsing approach is exactly what's needed to spot something that *isn't* normal
